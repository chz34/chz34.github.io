--- title: PyTorch FSDP2 全解析 date: 2026-04-02 00:00:00 +0800 categories: [AI 框架, 分布式训练] tags: [FSDP2, PyTorch, DTensor, torch.compile, 分布式训练] layout: post --- > 基于 PyTorch 主干源码（`torch/distributed/fsdp/_fully_shard/`）与 TorchTitan Llama3 训练实践，逐层拆解 FSDP2 的参数表示、通信调度、编译集成与多维并行组合机制。 ## 为什么需要 FSDP2

2022 年随 PyTorch 1.12 发布的 FSDP1（FullyShardedDataParallel）在工程上取得了成功，但其内部实现积累了相当的设计债务：参数被合并成 FlatParameter、通信通过 hook 动态插入、梯度需要从 flat buffer 重新拆解。这些决策在 torch.compile 普及之后暴露出严重的兼容性问题——hook 是图中断点，编译器无法穿越。

PyTorch 2.4 正式推出的 FSDP2（入口：torch.distributed.fsdp.fully_shard）从根本上重写了这套机制。它以 DTensor 作为参数表示的基础类型，以 Composable Distributed 框架统一并行语义，让整个训练迭代在单一计算图中完成——通信原语成为图的节点，而不是图的边界。

背景
Meta 内部的 TorchTitan（Llama3/Llama4 训练框架）已全面迁移到 FSDP2。本文所有源码分析基于 PyTorch 主干分支，关键路径为 torch/distributed/fsdp/_fully_shard/，共约 5000 行核心代码。

FSDP1 vs FSDP2：六个维度的根本差异

维度 FSDP1 FSDP2
API 入口 FullyShardedDataParallel(m)，包装类 fully_shard(m)，函数式原地修改
参数存储 FlatParameter：多参数合并为一个 1D 张量再切片 每个参数独立包成 DTensor(Shard(0))
通信调度 register_forward/backward_hook 在运行时插入 通信作为 FX graph 节点，编译器统一调度
梯度处理 flat grad → 拆解回各参数，易出 view 失效 梯度本身即 DTensor(Shard(0))，zero-copy view
torch.compile hook 边界导致图中断，无法整体优化 完整图捕获，通信与计算可跨边界融合
并行组合 与 TP/PP 组合繁琐，需手动协调 process group 共用 DeviceMesh + DTensor，原生 4D 并行

初始化链路：从 fully_shard 到 FSDPParam

fully_shard 用 @contract(state_cls=FSDPState) 装饰。这个装饰器来自 Composable Distributed 框架，它把一个 FSDPState 对象绑定到 module 上而不是替换 module 的类型——这是 FSDP2 能与其他并行方式组合的关键。初始化完毕后，再通过 _apply_to_module 动态创建新类并修改 module.__class__，将 FSDPModule 插入 MRO 最前端： 

# _fsdp_init.py · _apply_to_module
new_cls = type(
    f"FSDP{cls.__name__}",   # e.g. "FSDPTransformerBlock"
    (FSDPModule, cls),        # MRO: [FSDPTransformerBlock, FSDPModule, TransformerBlock, ...]
    {"__deepcopy__": unimplemented_deepcopy}
)
module.__class__ = new_cls   # 原地替换，isinstance(m, TransformerBlock) 仍然为 True

参数初始化的核心在 _init_param_group，它按 process group 把参数分组，每组创建独立的 FSDPParamGroup。这个分组机制支持 MoE 场景——专家参数可以用不同的 mesh（ep_mesh）而普通参数用 dp_mesh，两者在同一个 module 内共存。

关键设计
Lazy Init：FSDPState._lazy_init 在首次 forward 时才执行。此时才能确定哪个 state 是 root——root 的 reshard_after_forward 会被自动覆写为 False（保留完整参数以避免反向传播开头多一次 All-Gather），并把所有子状态的 _comm_ctx 替换为同一个 FSDPCommContext 对象，实现"全模型共享一套 CUDA stream"。

FSDPParam：参数的三种状态

每个 FSDPParam 维护一个状态机：

S
SHARDED — 正常态
module 上注册分片参数（DTensor, Shard(0)），storage 大小为 param_size/N。是唯一贡献参数显存的状态。
U
UNSHARDED — All-Gather 后
module 上注册完整参数。_unsharded_param 始终是同一个 Python 对象，storage 从 0 resize 到完整大小，保持 autograd 图的 alias 关系不断裂。
P
SHARDED_POST_FORWARD — 中间态
仅在 reshard_after_forward=int 时出现，reshard 到更小的 world_size（如节点内 8 卡），显存介于前两者之间。

Storage resize 是 FSDP2 解决 FSDP1 view 失效问题的核心手段。Free 时调用 free_storage(tensor) 把 storage 大小设为 0，Alloc 时调用 alloc_storage(tensor) 恢复，整个过程不改变张量对象的 Python identity，autograd 引擎持有的引用始终有效。

通信机制：双 Stream 流水线与可插拔后端

foreach_all_gather 的双 stream 设计

_fsdp_collectives.py 中的 foreach_all_gather 使用两个独立的 CUDA stream 形成流水线：

# foreach_all_gather：copy-in 和 AG 并行执行
with device_handle.stream(all_gather_copy_in_stream):
    # 把各参数分片 copy 到连续 buffer（_foreach_copy_ 批量操作）
    all_gather_input, all_gather_output = torch.ops.fsdp.all_gather_copy_in(
        all_gather_inputs, all_gather_output,
        inp_split_sizes, all_gather_input_numel, rank
    )

all_gather_stream.wait_stream(all_gather_copy_in_stream)

with device_handle.stream(all_gather_stream):  # 高优先级 stream
    all_gather_work = all_gather_comm(
        output_tensor=all_gather_output,
        input_tensor=all_gather_input,
        group=group, async_op=async_op
    )

all_gather_stream 被设置为高优先级，确保通信不被普通 compute kernel 抢占。copy-in 结束后 stream 间 event 同步，AG 在专用 stream 上异步发出，主 compute stream 可以继续执行当前层的后续操作。

三种 AllGather 实现

实现类分配方式适用场景
DefaultAllGather torch.empty（CUDA caching allocator） 通用默认路径
ProcessGroupAllocAllGather ProcessGroup 内置内存池 需要 backend 管理内存时
SymmMemAllGather CUDA symmetric memory pool NVLink 直连，NCCL Copy Engine zero-copy

SymmMemAllGather 调用 symm_mem.rendezvous(output_tensor) 让所有 rank 共享同一块物理内存地址空间，NCCL 可以启用 Copy Engine All-Gather——彻底绕开 PCIe，延迟和带宽均优于传统路径。

foreach_reduce 的 Reduce-Scatter 与梯度写回

反向传播完成后，foreach_reduce 在专用 reduce_scatter_stream 上执行：

# foreach_reduce 尾部：zero-copy 梯度写回
flat_grad_offset = 0
for padded_unsharded_size, fsdp_param in zip(padded_unsharded_sizes, fsdp_params):
    # as_strided 创建 view，不分配新内存
    new_sharded_grad = torch.as_strided(
        reduce_output,
        size=fsdp_param.sharded_size,
        stride=fsdp_param.contiguous_sharded_stride,
        storage_offset=flat_grad_offset,
    )
    # 包成 DTensor(Shard(0))，直接赋给 sharded_param.grad
    fsdp_param.sharded_param.grad = fsdp_param.to_sharded_dtensor(new_sharded_grad)
    flat_grad_offset += padded_unsharded_size.numel() // world_size

RS 输出是一个 1D flat buffer，所有参数的梯度通过 as_strided 以 zero-copy view 的形式指向这块 buffer，整个过程没有额外的内存分配。优化器的 fused=True 实现可以直接对这些 view 原地操作。

HSDP 路径
当 all_reduce_group is not None（即 2D mesh 的 HSDP 模式），foreach_reduce 在 RS 完成后还会额外走 all_reduce_stream 进行节点间梯度同步。两个 stream 之间用 event 串行，RS 和 AR 可以与其他层的反向计算重叠。

执行调度：单次迭代的完整轨迹

1
root_pre_forward _fsdp_state.py
迭代起点。all_gather_stream.wait_stream(current_stream) 确保 AG stream 在 compute stream 之后。若存在 post_optim_event（上一次迭代 optimizer 结束时记录），让 AG stream wait 该 event，彻底避免优化器写入和 All-Gather 读取的竞争。
2
pre_forward — All-Gather 触发 _fsdp_param_group.py
unshard(async_op) 发起 All-Gather，wait_for_unshard() 等待完成后调用 foreach_all_gather_copy_out，把 AG 输出拆分到各 FSDPParam._unsharded_param（storage resize）。同时通过 _states_to_forward_prefetch 提前发出下一层的 AG 请求。
3
forward 计算 用户模型 / torch.compile FX graph
参数以完整形态参与计算。AC（Activation Checkpointing）通过 selective_checkpoint_contexts 决定哪些 op 的输出需要保留。torch.compile 把整个 block 编译为 FX graph，FSDP 的 AG/RS 作为 graph 节点参与。
4
post_forward — reshard _fsdp_param_group.py
reshard() 根据策略决定是否释放完整参数（storage resize to 0）。_record_post_forward() 把当前 group 追加到 comm_ctx.post_forward_order，反向传播中按逆序预取。
5
pre_backward — 再次 All-Gather _fsdp_state.py
通过 output.register_hook(self._pre_backward) 触发，而非 module hook——这是 FSDP2 能与 torch.compile 协作的关键。若已 prefetch（unshard_async_op=True），则 unshard 为 no-op。
6
post_backward — Reduce-Scatter _fsdp_collectives.py
foreach_reduce 在 reduce_scatter_stream 上异步执行。RS 完成后通过 as_strided zero-copy 写回各参数的 DTensor grad。与下一层的前向计算流水线重叠。
7
root_post_backward_final_callback _fsdp_state.py
通过 Variable._execution_engine.queue_callback 在反向传播完成后执行清理：等待所有未完成的 RS event，清理 post_forward_order，若 is_last_backward=True 则调用 _finalize_backward。
8
optimizer.step() — 纯本地操作
每个 rank 的 sharded_param.grad 是 DTensor(Shard(0))，AdamW(fused=True) 直接操作。无任何跨 rank 通信。完成后可 record post_optim_event 供下次迭代使用。

TorchTitan 实践：Llama3 的 FSDP2 应用

apply_fsdp 的自底向上原则

torchtitan/models/llama3/parallelize.py 中的 apply_fsdp 展示了生产级使用模式。自底向上调用 fully_shard 是强制要求——先调用叶层，再调用 root，确保每个子模块的参数分配在 root 调用时已经稳定：

# torchtitan: apply_fsdp 调用顺序
# 1. 嵌入层（不 reshard，避免首 token 的 AG 开销）
fully_shard(model.tok_embeddings, **fsdp_config,
            reshard_after_forward=reshard_after_forward)

# 2. 每个 TransformerBlock（独立 FSDP unit，实现层间 overlap）
for layer_id, transformer_block in model.layers.items():
    fully_shard(transformer_block, **fsdp_config,
                reshard_after_forward=reshard_after_forward)

# 3. norm + output 组（False：保留在显存，反向传播立即需要）
fully_shard([model.norm, model.output], **fsdp_config,
            reshard_after_forward=False)

# 4. root model（必须最后）
fully_shard(model, **fsdp_config)

# 5. 关闭自动梯度归一化，由训练循环用 global token count 手动处理
disable_fsdp_gradient_division(model)

最后几层（norm + output）设 reshard_after_forward=False 是一个精心的优化：反向传播从输出层开始，这些参数会被立即用到，提前保留在显存里省去一次 AG 的延迟。

DeviceMesh 的 unflatten 拓扑

parallel_dims.py 中的 build_mesh 从单一 world mesh 通过 _unflatten 派生所有子 mesh，而非创建多个独立 ProcessGroup。这种设计避免了跨框架组件间的 process group 重复创建：

# ParallelDims.build_mesh() 示例（pp=2, dp_replicate=2, dp_shard=4, cp=1, tp=2）
world_mesh = init_device_mesh(device_type, (32,), mesh_dim_names=("world",))

dense_mesh = unflatten_mesh(
    world_mesh,
    ("pp", "dp_replicate", "fsdp", "tp"),
    (pp, dp_replicate, dp_shard * cp, tp),  # fsdp = dp_shard × cp
)

# fully_shard 使用的 mesh
dp_mesh = dense_mesh["dp_replicate", "fsdp"]  # 2D mesh → HSDP
# 或 dense_mesh["fsdp"]                   # 1D mesh → FSDP

值得注意的是，fsdp mesh 的大小是 dp_shard × cp——即使只启用了 Context Parallel（没有数据并行），FSDP 也会被激活，借用其 All-Gather/RS 机制来分布式管理模型权重的序列并行化。

并行应用的严格顺序

TP → CP → AC → torch.compile → FSDP 的应用顺序不是任意的。关键约束有两个：首先，AC 必须在 compile 之前，ptd_checkpoint_wrapper 需要在未编译状态下插入 hook；其次，compile 必须在 FSDP 之前，这样 Dynamo 能把 FSDP 的通信操作 trace 进图里，否则 FSDP hook 会成为图中断。

已知限制
MoE 模型中，GroupedExperts 的 FSDP 包装仍会产生图中断，因此 apply_compile_sparse 需要以更细粒度（子模块级别）调用 compile，并跳过 experts 部分。FSDP2 + compile + MoE 的三者完全无缝组合仍在开发中。

Selective AC：在显存与重计算之间的精确权衡

TorchTitan 的 _get_save_ops() 定义了一套 MUST_SAVE 算子集合：

类别算子理由
注意力 _scaled_dot_product_cudnn_attention, flex_attention 计算量大，重计算代价高
线性层 aten.linear（每隔一次） 交替保存：奇数次 SAVE，偶数次 RECOMPUTE
通信结果 reduce_scatter_tensor, all_to_all_single 重新通信代价远高于保存激活
编译输出 inductor_compiled_code compile 后的 fused kernel 不可单独重计算
D2H 拷贝 aten._to_copy (cuda→cpu) 跨设备传输结果不能重算（MoE metadata）

矩阵乘的交替策略（偶数次 RECOMPUTE）是一个典型的工程折中：Transformer block 中有两个主要的 matmul（attention proj 和 FFN），只保留其中一个的激活，显存减半而重计算开销翻倍但绝对值较小。

跨硬件使能：ProcessGroup 抽象的力量

FSDP2 的所有通信调用都通过 torch.distributed.ProcessGroup 抽象，切换硬件只需在 init_process_group 时指定 backend：

# NVIDIA GPU（默认路径，NVLink + InfiniBand）
dist.init_process_group(backend="nccl")
fully_shard(model, mesh=init_device_mesh("cuda", (world_size,)))

# AMD GPU（ROCm，RCCL 与 NCCL API 完全兼容）
dist.init_process_group(backend="nccl")  # PyTorch ROCm 版自动路由到 RCCL
fully_shard(model, mesh=init_device_mesh("cuda", (world_size,)))

# CPU / 调试
dist.init_process_group(backend="gloo")
fully_shard(model, mesh=init_device_mesh("cpu", (world_size,)),
            offload_policy=CPUOffloadPolicy())

# TPU / NPU（torch_xla）
dist.init_process_group(backend="xla")
fully_shard(model, mesh=init_device_mesh("xla", (world_size,)))
FSDPParam 的多设备支持
_fsdp_param.py 中的 fsdp.copy_ 自定义算子注册了 CUDA、XPU、HPU、CPU、MTIA 五个 dispatch key，确保参数 copy 在不同设备上都有正确的实现。all_gather_copy_in 同样如此，这是 FSDP2 能在华为 Ascend（HPU）等非 NVIDIA 设备上运行的基础。

总结：FSDP2 的设计哲学

通读 5000 行源码，FSDP2 的设计哲学可以归纳为三个层次的解耦：

表示与调度解耦。参数的分片语义（DTensor Placement）是参数自身的属性，不依赖外部包装类。调度逻辑（何时 AG、何时 RS）通过 state 对象和 hook 附着在 module 上，两者可以独立演化。

通信与计算解耦。All-Gather 和 Reduce-Scatter 通过独立的高优先级 stream 异步执行，通过 event 与 compute stream 同步。在 torch.compile 路径下，通信原语进一步成为图节点，编译器可以自由安排它们与计算 kernel 的相对顺序，实现真正的 overlap。

框架与硬件解耦。AllGather 和 ReduceScatter 是抽象接口，默认实现、ProcessGroup 内存池实现、SymmMem 实现可以随时切换，上层 FSDP 逻辑不感知差异。DeviceMesh 统一描述拓扑，切换硬件只需修改 backend 字符串。

这三层解耦共同支撑起一个能在 NVIDIA/AMD/Ascend 上运行、能与 TP/CP/PP 自由组合、能被 torch.compile 整体优化的现代分布式训练引擎。随着 FSDP2 逐步成为 PyTorch 的标准 DP 路径，理解这些机制将是高效使用下一代大模型训练基础设施的必要条件。
